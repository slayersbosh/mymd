# product-spec-builder

## 目的

不是让用户写需求文档扔给 AI，而是**反向**——AI 通过追问帮你把模糊想法变成结构化的 Product Spec。

## 输入

用户的一句话想法（模糊的）

## 输出

`PRODUCT-SPEC.md` — 面向 AI 的执行基准，不是给人看的文档

## 执行流程

### Step 1: 需求澄清追问

针对以下维度系统追问，直到没有遗漏：

**产品定位**
- 做什么产品？解决什么核心问题？
- 目标用户是谁？画像描述
- 竞品有哪些？差异化在哪里？

**功能范围**
- 核心功能（必须有的）
- 次要功能（最好有的）
- 边界功能（明确不支持的）
- MVP 功能优先级排序

**交互方式**
- 入口形式（Web/App/桌面/CLI？）
- 主要交互模式（对话/表单/编辑器/仪表盘？）
- 用户操作路径的主流程
- 异常流程如何处理

**数据与状态**
- 需要存储什么数据？
- 数据量级预估
- 是否需要账户系统？认证方式？
- 离线场景考虑？

**技术约束**
- 有没有特定技术栈要求？
- 有没有部署环境限制？
- 有没有性能要求？

**非功能需求**
- 安全性要求
- 可用性要求（99.9%？）
- 国际化/多语言？

### Step 2: 生成 PRODUCT-SPEC.md

```markdown
# [产品名称] — Product Spec

> 此文档是 AI 执行基准，不是给人阅读的文档。

## 1. 产品概述
[一句话描述]

## 2. 目标用户
[用户画像]

## 3. 功能规格

### 3.1 核心功能

#### Feature-001: [功能名]
- **描述**: 
- **输入**: 
- **输出**: 
- **用户故事**: 
- **验收标准**:
  - [ ] 
- **优先级**: P0

#### Feature-002: ...
```

## 验收标准

- Spec 覆盖用户 80% 以上的隐含需求（通过追问挖出）
- 每条 Feature 有明确的输入/输出/验收标准
- 没有歧义条款（"用户友好"之类的词全部量化）
- 功能优先级 P0/P1/P2 明确

## 迭代模式

需求变更时：
1. 更新 PRODUCT-SPEC.md 对应条目
2. 在 CHANGELOG.md 追加变更记录
3. 通知 design-brief-builder 重新评估视觉影响
