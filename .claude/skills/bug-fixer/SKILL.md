# bug-fixer

## 目的

不是看到报错就试着改，而是一套医生诊断式的调试流程。核心原则：**不猜不试**。

## 四阶段调试法

### 第一阶段：收集证据

按顺序执行，不跳过：

```
1. 读取完整的错误信息和 stack trace
2. 复现 bug，确认是稳定复现还是偶发
3. 检查最近的代码变更（git diff）
4. 追踪数据流：从触发点 → 报错点经过了哪些函数
```

**证据清单**：
- [ ] 完整的 error message
- [ ] stack trace（每一行）
- [ ] 复现步骤（最小步骤）
- [ ] 偶发率（100% / 50% / 10%）
- [ ] 相关代码变更记录
- [ ] 数据流路径

### 第二阶段：分析模式

```
1. 找到正常工作的相似功能做对比
2. 识别差异和可疑之处
3. 缩小范围到 1-2 个最可能的根因
```

**分析模板**：
```
正常功能 A：[代码片段或行为描述]
异常功能 B：[代码片段或行为描述]
差异点：[具体差异]
可疑之处：[列出 1-2 个假设]
```

### 第三阶段：假设验证

```
1. 基于证据形成最多 3 个假设（按可能性排序）
2. 用最小改动验证最可能的假设
3. 假设被否定 → 记录原因 → 换下一个
4. 不同时验证多个假设
```

**假设格式**：
```
假设 #1（可能性: 80%）:
  内容：bug 是因为 X 没有做 Y 校验
  验证方法：加一行 console.log 打印 X 的值
  预期：如果假设正确，X 应该为 undefined
  验证结果：[TBD]
```

### 第四阶段：实施修复

```
1. 一次只改一个逻辑点
2. 改完立即编译验证
3. 改完立即功能验证
4. 改完回归验证（确保修了 A 不会坏 B）
```

**回归验证清单**：
- [ ] 编译通过
- [ ] 原 bug 不再复现
- [ ] 相关功能仍然正常
- [ ] 单元测试全部通过

## 关键规则

### 不猜不试
- 没有证据不下结论
- 一次只改一处
- 改完立即验证

### 同一个 bug 反复修 3 次还不好
停下来，重新审视问题本身：
- 可能不是代码层面问题
- 可能是架构问题
- 可能是环境问题
- 可能是理解偏差

**不硬磕**。

## 输出格式

```markdown
# Bug 修复报告

## Bug ID
[自动生成: BUG-YYYYMMDD-NNN]

## 证据收集
- 错误信息: ...
- 复现步骤: ...
- 偶发率: ...

## 分析
- 正常功能对比: ...
- 差异点: ...
- 假设: ...

## 修复
- 修改文件: ...
- 修改内容: ...
- 验证结果: ...

## 回归测试
- [ ] 原 bug 不再复现
- [ ] 相关功能正常
```

## 与其他 Skill 的关系

- 被 code-review 调用（Stage 2 失败时）
- 修复后触发重新 Review
- 同一 bug 反复出现 → 触发 evolution-runner
